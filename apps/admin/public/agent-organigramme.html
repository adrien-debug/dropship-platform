Plan de relance

Ce document reformule la situation après les tests réels. Le sujet n'est plus "est-ce que tout marche ?". Le sujet est : quel produit on garde, qu'est-ce qu'on coupe, et dans quel ordre on répare.

Résumé exécutif

Le système n'est pas un agent autonome complet. En l'état, c'est un pipeline e-commerce partiellement fonctionnel entouré de couches qui hallucinent, ralentissent ou bloquent. Le plan n'est donc pas d'améliorer tout en parallèle, mais de resserrer le produit autour du pipeline qui exécute vraiment.

1. Ce qu'on essaye vraiment de construire

Vision produit

L'utilisateur donne une niche, un marché, un positionnement et éventuellement un design system. Ensuite le système doit trouver des produits cohérents, créer la boutique, générer le contenu, brancher le storefront et déployer sans intervention manuelle lourde.

Input attendu Niche produit, marché cible, gamme de prix, identité visuelle éventuelle.
Output attendu Une boutique live avec les bons produits, un branding cohérent et des pages utilisables.
Promesse minimale crédible Transformer une intention e-commerce en pipeline exécutable, pas en chat “magique”.

Architecture produit cible

1
Sourcing produits

Extraction et normalisation des bons keywords, recherche CJ, filtrage et sélection.

2
Import Medusa

Création du canal de vente et import des produits retenus.

3
Branding + contenu

Nom, angle, copy, hero, pages et design system alignés avec la niche.

4
Build + déploiement

Storefront compilé, vérifié, déployé, puis statut final mis à jour.

2. Ce que disent les tests, sans se mentir

Constat

Les tests ne disent pas que "tout est mauvais". Ils disent que le coeur pipeline existe, mais qu'il est entouré de mauvais abstractions. Le danger actuel vient surtout du décalage entre la promesse d'un agent complet et la réalité d'un pipeline incomplet.

Brique État réel Ce qui marche Ce qui casse Décision
Search pipeline utile Recherche CJ rapide, résultats pertinents sur keywords propres. Faible tolérance aux typos, phrases longues, langage oral. garder
CJ client solide Résultats rapides, infra stable côté sourcing. La qualité finale dépend trop du pré-traitement amont. garder
Content writer site partiel Peut produire une base de boutique crédible. La partie enrichissement est beaucoup trop lente. resserrer
Chat agent toxique Peut reformuler et produire du texte. N'exécute rien, hallucine, simule des actions inexistantes. retirer du flux
Debug-fix loop instable Essaye de corriger automatiquement. Boucle sur les mêmes erreurs, brûle du temps et du compute. désactiver
Marketing tools ambigu Peuvent générer un plan de campagne. Le retour ressemble à une action alors que ce n'est qu'un draft. renommer

3. Le vrai plan maintenant

Principe directeur

Arrêter de vendre un “agent tout faire”. Revenir à une promesse simple : pipeline e-commerce pilotée par intention. Chaque brique doit soit exécuter réellement, soit être sortie du chemin critique.

Ce qu'on garde comme coeur produit Search pipeline, CJ client, import catalogue, génération storefront contrôlée.
Ce qu'on retire du chemin critique Chat libre, debug-fix en boucle, sorties marketing ambiguës.
Ce qu'on ajoute avant tout Pré-traitement d'input pour absorber phrases, typos, oral et enums humains.

Ordre d'exécution recommandé

A
Réduire le périmètre

Le produit devient officiellement un pipeline assistée, pas un agent autonome.

B
Accélérer l'enrichissement

Batch + parallélisation pour faire tomber le temps d'attente perçu.

C
Normaliser l'input utilisateur

Traduction, correction, extraction de keywords, mapping marché/positioning.

D
Fallbacks fiables

Quand le codegen échoue, revenir à un template sûr au lieu de re-boucler.

4. Priorités des 2 prochains sprints

Priorité Objectif Impact utilisateur Pourquoi maintenant
P1 Sortir le chat agent du flux principal Réduit les promesses fausses et la confusion. C'est la source principale de perte de confiance.
P1 Paralléliser l'enrichissement produit Réduit fortement le temps d'attente de bout en bout. C'est le plus gros bottleneck mesuré.
P2 Ajouter un pré-processeur NLP simple Accepte enfin les vraies formulations utilisateur. Améliore le taux de succès sans refaire toute l'architecture.
P2 Remplacer la debug-fix loop par un fallback template Évite les boucles de 7 minutes pour zéro résultat. La stratégie actuelle prouve qu'elle ne converge pas.

5. Conclusion

Le plan, en une phrase

On ne cherche plus à sauver un faux agent généraliste. On consolide un pipeline e-commerce qui exécute vraiment, on supprime les couches trompeuses, puis on réintroduit de l'intelligence uniquement là où elle améliore le taux de succès réel.

Base de lecture : 28 tests live exécutés le 5 avril 2026, services disponibles, résultats observés sur exécution réelle. Ce document sert à réaligner la roadmap, pas à embellir les résultats.
